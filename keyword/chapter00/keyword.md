### PK, FK란?

  **PK (primary key : 기본키) : 테이블에서 각 행의 정보들을 식별할 수 있는 정보**

  - 테이블 행의 여러 정보들 중 행을 식별할 수 있어야 하기 때문에 NOT NULL, Unique 해야함.
  - 유일성과 최소성을 가짐
  - 유일성 : 여러 개의 행이 존재할 때 각각의 행을 유일하게 식별할 수 있는 것
  - 최소성 : 키를 구성하는 속성들 중 꼭 필요한 최소한의 속성들로만 키를 구성

  **FK ( foreign key : 외래키) : 참조하는 테이블과 참조되는 테이블의 관계**

  - 하나의 테이블에서 다른 테이블의 PK를 참조한다
  - 자기참조 : FK는 자기 자신의 기본키를 참조할 수 있다 (대댓글 기능 구현 위함)

###  ERD란?

  **ERD(Entity-Relationship Diagram) : 데이터베이스 설계에서 사용되는 그림**

  - Entity(엔티티) : 시스템이 관리하고자 하는 ‘대상’ , 테이블의 이름
    - 강한 엔티티 : 기본키가 있는 객체
    - 약한 엔티티 : 별도의 기본 키 없이 다른 개체의 키를 이용하는 객체
    - 예시 : 회원, 게시글, 댓글
  - Attribute(속성) : 엔티티가 가진 ‘상세 정보’
    - 예시 : 회원의 이름, 아이디
  - Relationship(관계) : 엔티티와 엔티티 사이의 ‘연관성’
    - 1:1 , 1:N , N:M 관계들
    - 예시 : 회원이 게시글을 ‘작성한다’

### 연관관계란? 그리고 연관관계를 설정하는 방법은?

  **연관관계 : 두 개 이상의 테이블이 서로 어떻게 연결되어 있는지를 정의하는 것**

  1. 연관관계의 종류
   - 1:1 (일대일) : 한 사용자는 하나의 프로필만 가질 수 있는 경우
   - 1:N (일대다) : 한 게시글에 여러 개의 댓글이 달리는 경우
   - N:M (다대다) : 학생 여러 명이 여러 과목을 수강하고, 한 과목을 여러 학생이 듣는 경우
        - 워크북 설명처럼 다대다는 각각 1:N으로 풀어서 설계한다

  **연관관계 설정 방법**

  1. **데이터 베이스**
   - 외래 키를 사용하여 테이블 간의 연결 고리를 만든다
   - N쪽 테이블에 1쪽의 기본 키를 저장하는 컬럼 만든다
   - 게시글 테이블과 댓글 테이블 1 : N 관계

### 정규화란?

  **정규화** : 데이터의 중복을 없애고 구조를 깔끔하게 다듬는 과정

  → 정규화를 제대로 하지 않으면 Anomaly (이상현상)이 발생할 수 있다

  - 삽입 이상 : 데이터를 넣고 싶은데 불필요한 정보까지 함께 넣어야하는 상황
  - 갱신 이상 : 한 곳의 정보를 바꿨는데 다른 곳에 남은 중복 데이터와 불일치 상황
  - 삭제 이상 : 특정 정보를 지웠는데 다른 정보까지 같이 지워지는 상황

  **정규화 단계**

  - 제1정규형 (1NF): 원자 값 (Atomic Value)
    - **규칙:** 각 칸(셀)에는 딱 하나의 값만 들어가야 합니다.
    - **상황:** 만약 '취미' 컬럼에 축구, 농구, 야구 라고 리스트로 들어가 있다면?
    - 한 칸에 하나씩만 들어가도록 쪼개야 합니다.

  - 제2정규형 (2NF): 부분 함수 종속 제거
    - **규칙:** 기본 키(PK)가 복합키(여러 컬럼의 조합)일 때, PK의 일부분에만 의존하는 컬럼이 있어서는 안 됩니다.
    - **상황:** 학번, 과목 코드가 PK인데, 이름 컬럼이 있다면? → 이름은 학번에만 의존하므로 별도 테이블(학생 테이블)로 분리해야 합니다.

  - 제3정규형 (3NF): 이행적 함수 종속 제거
    - **규칙:** 기본 키가 아닌 컬럼들끼리 서로 의존 관계가 있어서는 안 됩니다. (A -> B -> C 관계 제거)
    - **상황:** 학번이 PK인데, 학과ID를 통해 학과이름을 알 수 있다면? → 학과이름은 학번이 아니라 학과 ID에 종속된 것이므로 별도의 학과 테이블로 분리합니다.

  **장단점**

  - 장점 : 데이터의 무결성이 높아지고 저장공간을 효율적으로 쓴다. 관리가 편해진다
  - 단점 : 테이블이 자꾸 쪼개지기 때문에 데이터를 가져올 때 조인을 많이 해야한다.

### 반 정규화란?

  **반정규화** : 정규화된 데이터베이스의 성능을 향상시키기 위해 의도적으로 정규화 원칙을 어기고 데이터 중복을 허용하거나 테이블을 합치는 과정 (조회 성능의 최적화)

  **반정규화 기법**

  - 테이블 합치기 : 1:1 관계나 1:N 관계의 테이블을 하나로 합쳐서 조인을 아예 없애버리는 방법
  - 중복 컬럼 추가 : 자주 조회되는 컬럼을 다른 테이블에도 중복해서 넣어두는 방식
  - 파생 컬럼 추가 : 계산된 값을 미리 저장해두는 방식입니다.
      - 예시: '게시글' 테이블에 '댓글 수' 컬럼을 추가
  - 테이블 수평/수직 분할
      - 수평 분할 : 데이터가 너무 많을 때 행을 기준으로 쪼개기
      - 수직 분할 : 한 테이블에 컬럼이 너무 많을 때 자주 쓰는 컬럼과 안 쓰는 컬럼 분리


  **장단점**
    
  - 장점 : 조회 속도 빨라짐 , 쿼리 단순해짐, 사용자 편의성 높아짐
  - 단점 : 입력/수정/삭제 속도 느려짐, 데이터 중복 발생, 데이터 불일치 위험 큼

### DB에서의 상속 관계 표현은 어떻게 하는가?

  **상속 구조를 테이블로 변환하는 세 가지 전략 사용**

  1. 조인 전략 : 부모 클래스와 자식 클래스를 각각 별도의 테이블로 만들고 데이터를 조회할 때 조인(Join)을 사용하는 방식입니다.
     - 구조 : 부모 테이블의 PK를 자식 테이블의 PK이자 FK로 가진다
     - 장점 : 테이블이 정규화되어 있고 저장 공간을 효율적으로 사용한다
     - 단점 : 조회할 때 조인이 많이 발생하여 성능이 저하될 수 있고 쿼리가 복잡하다

       ```
       @Entity
       @Inheritance(strategy = InheritanceType.JOINED)
       @DiscriminatorColumn(name = "DTYPE") // 부모 테이블에서 자식 구분용 컬럼
       public abstract class Item {
           @Id @GeneratedValue
           private Long id;
           private String name;
       }
        
       @Entity
       public class Album extends Item {
           private String artist;
       }
       ```

  2. 단일 테이블 전략 : 부모와 자식의 모든 속성을 하나의 테이블에 넣는 방식
      - 구조: 어떤 자식 타입인지 구분하기 위한 구분컬럼이 반드시 필요합니다.
      - 장점: 조인이 필요 없어 조회 성능이 가장 빠르고 쿼리가 단순합니다.
      - 단점: 자식 클래스가 가진 컬럼들은 모두 NULL을 허용해야 하며 테이블이 너무 커지면 오히려 성능이 나빠질 수 있습니다.

      ```
      @Entity
      @Inheritance(strategy = InheritanceType.SINGLE_TABLE)
      @DiscriminatorColumn(name = "DTYPE")
      public abstract class Item {
          @Id @GeneratedValue
          private Long id;
          private String name;
      }
      ```

  3. 구현 클래스마다 테이블 전략 : 부모 테이블 없이 자식 클래스마다 독립적인 테이블 만드는 방식
      - 구조: 자식1 테이블, 자식2 테이블이 각각 모든 정보를 다 가집니다.
      - 장점: 자식 타입을 명확하게 구분해서 처리할 때 좋습니다.
      - 단점: 여러 자식 테이블을 한꺼번에 조회할 때(부모 타입으로 조회할 때) UNION을 써야 해서 성능이 매우 안 좋습니다.

      ```
      @Entity
      @Inheritance(strategy = InheritanceType.TABLE_PER_CLASS)
      public abstract class Item {
          @Id @GeneratedValue
          private Long id;
          private String name;
      }
      ```

### 인덱스란?

  **인덱스** : 조회 성능을 획기적으로 높이기 위해 사용

  - 데이터의 양이 많을 때나 WHERE이나 JOIN에 자주 사용되는 컬럼에 대부분 사용

  **인덱스 작동원리** : B- Tree 자료구조 이용하여 인덱스 관리

  - 정렬 상태 유지: 인덱스는 특정 컬럼의 값을 기준으로 항상 정렬된 상태를 유지합니다.
  - 이진 탐색과 유사: 정렬되어 있기 때문에 "중간값보다 큰가, 작은가?"를 따지며 데이터를 찾아가므로 처음부터 끝까지 다 뒤지는 것보다 훨씬 적은 단계만으로 원하는 데이터를 찾을 수 있습니다.

  **장단점**

  - 장점 :  WHERE절을 이용한 검색과 JOIN, ORDER BY 작업 속도가 빨라짐, 시스템 부하 감소
  - 저장 공간 차지, CUD(입력, 수정, 삭제) 성능 저하
